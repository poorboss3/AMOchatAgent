# AI Agent Chat 技术方案文档

## 1. 项目概述

本项目构建一个移动端 AI Agent 对话系统，用户通过手机浏览器打开 Chat 页面，与 LLM 进行多轮对话，最终由后端 LLM Agent 调用第三方 API 完成订单创建。

---

## 3. 技术选型

### 3.1 前端

| 模块 | 技术选型 | 说明 |
|------|---------|------|
| 框架 | Vue 3 + Vite（或纯 HTML） | 轻量，移动端友好；纯 HTML 方案无需构建工具 |
| UI 样式 | Tailwind CSS | 移动端适配方便，按需裁剪体积小 |
| 通信方式 | Fetch API + SSE（Server-Sent Events） | 支持流式输出，提升对话体验 |
| 部署 | 静态文件 / Nginx / CDN | 可独立于后端单独部署 |

**推荐方案**：使用 **纯 HTML + Tailwind CDN + 原生 JS**，零构建依赖，独立部署一个 HTML 文件即可运行。

### 3.2 后端

| 模块 | 技术选型 | 说明 |
|------|---------|------|
| 框架 | ASP.NET Core 8 Web API | 跨平台，性能优秀 |
| LLM 集成 | Semantic Kernel 或 官方 HTTP Client | 支持 Tool Calling / Function Calling |
| 会话管理 | 内存缓存（IMemoryCache）或 Redis | 多轮对话上下文存储 |
| HTTP 客户端 | IHttpClientFactory + Polly | 调用第三方 API，支持重试、超时 |
| 日志 | Serilog | 结构化日志，便于排查验证失败问题 |
| 序列化 | System.Text.Json | .NET 内置，性能好 |

## 4. 前端设计

### 4.1 页面结构

```
┌─────────────────────────┐
│       顶部标题栏         │  固定高度，品牌名/标题
├─────────────────────────┤
│                         │
│       消息气泡区域       │  flex-col，消息向上堆叠
│   ┌─────────────────┐   │  用户消息靠右，AI消息靠左
│   │  AI: 请提供...  │   │
│   └─────────────────┘   │
│          ┌──────────┐   │
│          │ 用户消息 │   │
│          └──────────┘   │
│   ┌─────────────────┐   │
│   │  AI: 正在处理... │  │  流式输出，打字机效果
│   └─────────────────┘   │
│                         │
├─────────────────────────┤
│  [输入框]       [发送]   │  固定在底部，适配软键盘弹出
└─────────────────────────┘
```

### 4.2 移动端适配要点

- `<meta name="viewport" content="width=device-width, initial-scale=1">` 必须设置
- 输入框固定在底部（`position: fixed; bottom: 0`），配合 `padding-bottom` 适配软键盘
- 消息区域使用 `overflow-y: auto`，保持滚动至最新消息
- 触控友好：按钮最小点击区域 44×44px
- 禁止页面整体缩放（`user-scalable=no`）

### 4.3 会话标识

- 前端初始化时生成 `sessionId`（UUID），存于 `sessionStorage`
- 每次请求携带该 `sessionId`，后端据此关联多轮上下文

### 4.4 通信方式

- **普通请求**：`POST /api/chat`，返回 JSON
- **流式输出**（推荐）：`POST /api/chat/stream`，后端返回 `text/event-stream`，前端使用 `EventSource` 或 `fetch` + `ReadableStream` 接收，实现打字机效果

## 5. 后端 API 设计

### 5.1 接口定义

#### POST /api/chat

**请求体**
```json
{
  "sessionId": "uuid-string",
  "message": "我想订购商品X，数量3件"
}
```

**响应体（非流式）**
```json
{
  "sessionId": "uuid-string",
  "reply": "好的，请提供您的收货地址",
  "status": "collecting_info" | "order_created" | "error",
  "orderResult": null | { "orderId": "xxx", "amount": 99.0 }
}
```

#### POST /api/chat/stream

响应类型：`text/event-stream`

```
data: {"type":"delta","content":"好的，"}
data: {"type":"delta","content":"请提供"}
data: {"type":"done","status":"collecting_info","orderResult":null}
```

#### DELETE /api/chat/{sessionId}

清除会话上下文（用户主动结束对话时调用）。

## 6. LLM Agent 设计

### 6.1 核心思路：Tool Calling 模式

使用 LLM 的 **Function Calling / Tool Calling** 能力，将订单创建包装为一个工具（Tool）。

```
用户消息 → LLM 判断意图
    │
    ├─ 信息不足 → LLM 生成追问 → 返回前端 → 用户补充
    │
    └─ 信息足够 → LLM 生成 Tool Call：create_order(params)
                        │
                        └─ 后端执行 OrderApiClient.CreateOrder()
                                │
                                ├─ 成功 → 将结果注入对话 → LLM 生成确认话术
                                │
                                └─ 失败（验证不通过）→ 将错误信息注入对话
                                        │
                                        └─ LLM 解析错误 → 生成针对性追问 → 继续多轮
```

### 6.2 支持的 Tool 定义示例

```
Tool: create_order
参数:
  - product_id: string (必填) - 商品ID
  - quantity: integer (必填) - 数量
  - receiver_name: string (必填) - 收货人姓名
  - receiver_phone: string (必填) - 收货人手机号
  - receiver_address: string (必填) - 收货地址
```

### 6.3 对话上下文管理

每个 `sessionId` 对应一个 `ConversationContext`，包含：

| 字段 | 说明 |
|------|------|
| `Messages` | 完整对话历史（含 system、user、assistant、tool 消息） |
| `CollectedParams` | 已收集到的订单参数（字典） |
| `AttemptCount` | 当前尝试次数 |
| `CreatedAt` | 创建时间（用于过期清理） |

上下文存储于后端 `IMemoryCache`，设置 **滑动过期时间**（如 30 分钟无操作自动清除）。

### 6.4 System Prompt 设计原则

```
你是一个订单助手，负责帮助用户创建订单。
- 对话语言：中文
- 你需要收集以下信息才能创建订单：商品ID、数量、收货人姓名、手机号、地址
- 信息不足时，请礼貌地向用户逐步追问，不要一次性罗列所有问题
- 当所有信息收集完毕后，调用 create_order 工具
- 如果 create_order 返回错误，根据错误信息向用户说明原因并引导重新提供正确信息
- 不要编造订单结果，所有订单操作必须通过工具完成
```

## 7. 多轮对话与验证失败处理

### 7.1 典型验证失败场景

| 场景 | 第三方 API 返回 | LLM 处理策略 |
|------|----------------|--------------|
| 手机号格式错误 | `{"code":"INVALID_PHONE"}` | 提示用户重新输入手机号 |
| 商品库存不足 | `{"code":"OUT_OF_STOCK"}` | 告知用户库存不足，询问是否更换商品或数量 |
| 地址无法识别 | `{"code":"ADDRESS_NOT_FOUND"}` | 引导用户提供更详细地址 |
| 用户未实名认证 | `{"code":"KYC_REQUIRED"}` | 告知需要实名认证，引导完成后重试 |
| 网络超时 | 请求超时异常 | 告知用户网络异常，询问是否重试 |

### 7.2 错误注入对话流

第三方 API 返回错误后，后端将错误信息以 `tool` 角色消息写入对话历史，格式如：

```json
{
  "role": "tool",
  "content": "{\"success\":false,\"errorCode\":\"INVALID_PHONE\",\"message\":\"手机号格式不正确\"}",
  "tool_call_id": "xxx"
}
```

LLM 接收到此消息后，自动生成用户友好的追问，实现无缝多轮处理。

### 7.3 最大重试保护

设置 `MaxAttempts`（如 5 次），超过后由 LLM 告知用户本次无法完成订单，建议人工客服介入，防止无限循环。

## 8. 部署方案

### 8.1 独立部署架构

```
┌──────────────────┐     ┌──────────────────────────┐
│   前端静态文件    │     │      后端 .NET Core       │
│  (Nginx / CDN)   │────▶│   (Docker / Linux 服务)   │
│  index.html      │     │   监听 :8080              │
│  端口: 80/443    │     └──────────────┬────────────┘
└──────────────────┘                    │
                                ┌───────┴────────┐
                                ▼                ▼
                         ┌─────────────┐  ┌──────────────┐
                         │  LLM API    │  │ 第三方订单API │
                         └─────────────┘  └──────────────┘
```

### 8.2 前端部署

- 编译产物（或单 HTML 文件）部署至 Nginx / 对象存储 / CDN
- 通过反向代理将 `/api/*` 请求转发至后端，解决跨域问题
- 或后端开启 CORS，前端通过环境变量配置 API Base URL

### 8.3 后端部署

- 使用 `dotnet publish` 发布为独立可执行文件
- 通过 Docker 容器化，`Dockerfile` 基于 `mcr.microsoft.com/dotnet/aspnet:8.0`
- 配置文件通过环境变量注入（LLM API Key、第三方 API 地址等），不硬编码

### 8.4 配置项（appsettings.json / 环境变量）

| 配置项 | 说明 |
|--------|------|
| `Llm:ApiKey` | LLM 服务 API Key |
| `Llm:BaseUrl` | LLM API 地址（支持 OpenAI 兼容接口） |
| `Llm:Model` | 使用的模型名称 |
| `OrderApi:BaseUrl` | 第三方订单 API 地址 |
| `OrderApi:ApiKey` | 第三方 API 认证 Key |
| `Session:ExpireMinutes` | 会话过期时间，默认 30 |
| `Session:MaxAttempts` | 最大尝试次数，默认 5 |

## 9. 安全考虑

| 风险点 | 应对措施 |
|--------|---------|
| LLM API Key 泄露 | 仅在后端持有，前端无任何密钥 |
| 第三方 API Key 泄露 | 同上，后端持有 |
| 接口滥用 | 后端对 `/api/chat` 添加频率限制（Rate Limiting），基于 IP 或 sessionId |
| Prompt Injection | System Prompt 设计时明确角色边界；对用户输入做长度截断 |
| 敏感数据存储 | 会话上下文仅存内存/Redis，不落地持久化数据库 |
| HTTPS | 生产环境强制 HTTPS |

## 10. 扩展性考虑

| 扩展方向 | 说明 |
|----------|------|
| 多 Tool 支持 | 可按需增加更多 Tool（查询订单、取消订单等），LLM 自动路由 |
| 多 LLM 支持 | `ILlmAgentService` 抽象层可切换不同 LLM 提供商 |
| 会话持久化 | 将 `IMemoryCache` 替换为 Redis，支持服务重启后恢复会话 |
| 多语言 | System Prompt 支持按用户 locale 切换语言 |
| 用户认证 | 后续可在 Chat API 前增加 JWT 认证，关联用户身份 |

## 11. 技术方案总结

| 层次 | 技术选型 | 关键决策原因 |
|------|---------|-------------|
| 前端 | 纯 HTML + Tailwind CDN + 原生 JS | 零构建依赖，单文件独立部署，移动端优先 |
| 通信 | SSE 流式输出 | 提升对话体验，避免等待感 |
| 后端框架 | ASP.NET Core 8 Web API | 满足 C# .NET Core 要求，性能强，生态成熟 |
| LLM 编排 | Tool Calling 模式 | LLM 自主决策何时调用订单 API，处理验证失败更灵活 |
| 上下文 | IMemoryCache + sessionId | 轻量，无外部依赖，满足多轮对话需求 |
| 第三方 API | IHttpClientFactory + Polly | 统一 HTTP 管理，支持重试与超时 |
| 部署 | Docker + Nginx | 独立部署，前后端分离，易于水平扩展 |
